# 图片表格转数据规则

## 核心原则

渲染脚本只接收结构化数据，不接收“写死在 Python 里的商品配置”。图片识别、人工校对、表格修正都必须落到标准商品 JSON，然后由通用渲染脚本读取。

## 标准数据文件

GPT 视觉识别生成的数据固定保存到：

```text
scripts/output/product_data/<商品文件夹名>-product_data-<时间戳>.json
```

例如：

```text
scripts/output/product_data/KQ26144 驭途者机能裤-product_data-20260617-103012.json
```

渲染脚本默认从 `scripts/output/product_data/` 读取每个商品最新时间戳版本。为了兼容旧流程，也可以在每个商品目录内放 `product_data.json`，或用 `--data-dir` 指定外部数据目录。

## JSON 结构

```json
{
  "info": {
    "名称": "",
    "货号": "",
    "材质": "",
    "颜色": "",
    "尺码": ""
  },
  "colors": ["颜色1", "颜色2"],
  "size_table": {
    "headers": ["尺码", "衣长", "胸围"],
    "rows": [
      ["110", "46", "74"],
      ["120", "49", "78"]
    ]
  },
  "quick_size": {
    "columns": ["15-19", "19-23"],
    "rows": ["100-110", "110-120"],
    "data": [
      ["110", "110"],
      ["", "120"]
    ]
  }
}
```

完整字段约束见：

```text
references/12-商品数据结构.md
```

说明：

- `colors` 数组用于保存拆分后的颜色顺序。
- `info.颜色` 可以保留源图识别到的原始颜色文本，但产品信息页渲染时会按颜色拆分并统一用 `/` 输出。
- 若同时存在 `colors` 和 `info.颜色`，产品信息页优先使用 `colors` 数组。
- `info.材质` 需要保留层级结构。若源图材质本身分为多项层级，例如 `面层/底层`、`面料一/面料二`、`里料/填充物` 等，识别成 JSON 时必须在层级之间写入换行符 `\n`，例如 `"面层：棉/聚酯纤维\n底层：氨纶"`。普通材质成分里的 `/`、顿号、逗号不代表换行。

## 图片识别流程

缺少 `product_data.json` 时，第一步必须使用当前对话中的 GPT 图像识别能力处理源图，不先进入渲染脚本。脚本只能消费结构化 JSON，不能替代视觉识别。

图片表格识别固定成三段：

1. GPT 视觉识别：从 `产品信息`、`选尺码`、`尺码快选` 图片读取字段、表格行列、空值 `/`、小数、百分号和区间。
2. 结构化：把识别结果整理成商品 JSON，保存到 `scripts/output/product_data/<商品文件夹名>-product_data-<时间戳>.json`，字段必须符合 `references/12-商品数据结构.md`。
3. 校验：运行 `scripts/validate_product_data.py`，通过后再渲染。

以后不能把某款商品的识别结果直接写进渲染脚本。

## 固定转换交付物

无论使用 OCR、视觉模型、Excel、CSV 还是人工校对，最终优先交付固定归档 JSON：

- `scripts/output/product_data/<商品文件夹名>-product_data-<时间戳>.json`

兼容交付物：

- 每个商品目录内的 `product_data.json`。
- 或 `--data-dir` 中与商品文件夹同名的 `<product-folder>.json`。

渲染脚本只读取这些 JSON。固定归档目录用于保留历史识别结果，重跑时默认取同一商品最新时间戳版本。

## 识别方法选择

优先级：

1. 如果商品目录缺少 `product_data.json`，优先使用 GPT 视觉模型直接识别图片并输出 JSON。
2. 如果用户能提供 Excel、CSV 或后台导出的结构化表，可用结构化表生成 JSON，并用 GPT 视觉抽查源图。
3. OCR 只作为辅助校验，不作为主流程；OCR 结果必须人工或程序校验后再进入渲染。

说明：

- 纯图片表格没有 100% 稳定 OCR，尤其低分辨率中文和细线表格。
- GPT 视觉模型更适合保持产品信息、尺码表、快选矩阵之间的行列关系；识别不确定时必须写入 `warnings`。
- 稳定批量的关键不是“每次写代码”，而是固定数据协议和校验流程。
- 识别可以换工具，但输出必须是同一个 `product_data.json`。

## 校验命令

校验单个文件：

```powershell
uv run python scripts/validate_product_data.py path/to/product_data.json
```

校验整个输入根目录：

```powershell
uv run python scripts/validate_product_data.py path/to/input-root
```

## 生成空模板

为每个商品生成空 `product_data.json`：

```powershell
uv run python scripts/create_product_data_template.py --input-root path/to/input-root
```

生成后填入识别结果，再运行校验。空模板只是占位，不代表数据已完成。

## 空值

空值仍按统一规则处理：

```text
""
"/"
"--"
"无"
"暂无"
"无数据"
```

产品信息空值留空；表格单格空值留空；整列全空由渲染层删除。

## 质量控制

识别完必须检查：

- 产品信息字段是否放到正确 key。
- 材质是否已在 JSON 阶段保留应有层级换行；渲染脚本只按 `info.材质` 中的显式换行分行，不猜测 `面层`、`底层`、`面料一` 等标签。
- 多个颜色是否已拆分到 `colors` 数组，产品信息页显示时会用 `/` 分割。
- 尺码表 headers 与每行数据列数是否匹配。
- 快选表 columns、rows、data 矩阵尺寸是否匹配。
- 尺码值是否被误识别，例如 `110` 不能变成 `ll0`。
- 小数、区间、百分号要保留原样。
